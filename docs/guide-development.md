你好！欢迎开始 E-Kanban 项目开发。本文档是**一步一步的引导**，你跟着做就行了。

## 🌐 网状知识库导航

本项目采用**网状知识库**结构，让你能够快速定位所需信息，而不是线性阅读所有文档。

### 核心知识库索引

| 知识库 | 文档 | 用途 | 使用场景 |
|--------|------|------|---------|
| **模块索引** | [kb-module-index.md](./kb-module-index.md) | 查找功能对应的代码位置 | 不知道代码放哪里？查这里 |
| **代码资产** | [kb-code-assets.md](./kb-code-assets.md) | 了解项目结构和已有代码 | 了解项目整体情况 |
| **问题库** | [kb-problem-solutions.md](./kb-problem-solutions.md) | 查找问题和解决方案 | 遇到问题？先查这里 |
| **代码模式** | [kb-code-patterns.md](./kb-code-patterns.md) | 参考代码模板和规范 | 写代码时参考模板 |
| **需求文档** | [src/feature-records/](../src/feature-records/) | 查看每个需求的详细信息 | 了解具体需求 |

### 快速导航指南

**场景 1：接到新需求**
```
1. 查需求详情 → src/feature-records/需求名称/01-需求分析.md
2. 查代码位置 → kb-module-index.md
3. 参考代码规范 → kb-code-patterns.md
4. 开始开发 → 继续阅读本文档
```

**场景 2：开发遇到问题**
```
1. 查问题库 → kb-problem-solutions.md
2. 查相似模块 → kb-module-index.md
3. 查代码模式 → kb-code-patterns.md
```

**场景 3：不知道代码放哪里**
```
直接查 → kb-module-index.md
```

> 💡 **提示**：建议将 [INDEX.md](./INDEX.md) 加入书签，它是知识库的总入口。

## 🚀 开始开发（跟着开发计划走）

### 阶段一：需求收集与分析

#### 步骤一：检查项目本地代码资产盘点报告

**为什么要做这一步？**
- 了解现有代码结构，避免重复造轮子
- 识别可复用的组件和服务
- 确保新需求与现有系统兼容

**怎么做？**
1. 打开 `docs/kb-code-assets.md`
2. 仔细阅读以下内容：
   - 项目整体架构
   - 现有模块划分
   - 已有的实体、服务、仓储、控制器
   - 技术栈和依赖关系
3. 记录下可以复用的模块和需要新增的模块

#### 步骤二：创建需求文件夹

**为什么要做这一步？**
- 保持文档结构清晰
- 便于后续维护和查找
- 形成标准化的需求管理流程

**怎么做？**
1. 在 `src/feature-records/` 目录下创建以需求名称命名的文件夹
2. 文件夹命名规范：使用小写字母和连字符，如 `copilot-project-path`、`manual-kanban-creation`
3. 创建以下文件结构：
   ```
   src/feature-records/需求名称/
   ├── README.md              # 需求概述和文档导航
   ├── 01-需求分析.md         # 功能需求分析和技术实现要点
   ├── 02-后端实现方案.md     # 后端技术实现细节
   ├── 03-前端实现方案.md     # 前端技术实现细节
   ├── 04-实施步骤.md         # 具体实施流程和部署步骤
   ├── 05-测试计划.md         # 测试方案与用例
   ├── 06-知识沉淀.md         # 开发经验和问题解决记录
   └── 07-预期效果.md         # 功能预期与验收标准
   ```

**GitHub Copilot 编程模式提示词模板**（创建需求文件夹）：

```bash
copilot -p "
我正在开发 E-Kanban 项目，需要创建一个新的需求文档。

需求名称：【需求名称】
需求描述：【需求描述】

请帮我：
1. 在 src/feature-records/ 目录下创建需求文件夹
2. 生成 README.md 文件，包含需求概述和文档导航
3. 生成 01-需求分析.md 文件模板
4. 生成其他必要的文档模板

请遵循现有需求文档的风格和结构。
" --allow-all-tools
```

#### 步骤三：编写需求分析文档

**怎么做？**
1. 编写 `01-需求分析.md`，包括：
   - 功能需求分析
   - 非功能需求
   - 技术实现要点
   - 数据模型设计
   - 业务流程
   - 风险评估

2. 编写 `02-后端实现方案.md` 和 `03-前端实现方案.md`

3. 编写 `04-实施步骤.md`、`05-测试计划.md`、`07-预期效果.md`

**GitHub Copilot 编程模式提示词模板**（编写需求分析）：

```bash
copilot -p "
我正在开发 E-Kanban 项目，需要编写需求分析文档。

现有代码资产：
$(cat docs/kb-code-assets.md)

需求描述：
【需求描述】

请帮我生成 01-需求分析.md 文档，包括：
1. 功能需求分析
2. 非功能需求
3. 技术实现要点
4. 数据模型设计
5. 业务流程
6. 风险评估

请确保与现有代码资产兼容。
" --allow-all-tools
```

#### 步骤四：需求评审与确认

**怎么做？**
1. 组织需求评审会议
2. 确认需求的完整性和可行性
3. 评估技术风险和实施难度
4. 更新 `README.md` 文档，说明需求概述
5. 提交需求文档到 Git 仓库
6. 创建需求开发分支

### 阶段二：开发实施

#### 按批次开发

打开 `plan-development.md`，找到**阶段二：开发实施**，按批次顺序做：

**每个任务的执行步骤**：

```
步骤 1：理解需求
  → 打开 src/feature-records/需求名称/01-需求分析.md
  → 理解功能需求和技术要点
  → 如果不清晰，查看相似需求的实现

步骤 2：定位代码位置
  → 打开 kb-module-index.md
  → 查找相似功能的代码位置
  → 确定新代码应该放在哪里

步骤 3：参考代码规范
  → 打开 kb-code-patterns.md
  → 找到对应的代码模板（实体、服务、控制器等）
  → 理解命名规范和代码风格

步骤 4：查看相似模块（重要！）
  → 在 kb-module-index.md 中找到相似功能
  → 阅读相似模块的代码实现
  → 复用相似的模式和逻辑

步骤 5：生成代码
  → 使用 Copilot 提示词模板
  → 提供足够的上下文（需求、规范、相似代码）
  → 生成代码

步骤 6：自验
  → 对照检查清单检查
  → Copilot 静态代码检查
  → 确保编译通过

步骤 7：更新状态
  → 在 plan-development.md 里把这个任务状态改成 ✅ 完成
  → 进入下一个任务
```

**关键提示**：
- ⚠️ 不要跳过"步骤 4：查看相似模块"，这是保证代码质量的关键
- 💡 如果遇到问题，先查 [kb-problem-solutions.md](./kb-problem-solutions.md) 看是否有类似问题

**GitHub Copilot 编程模式提示词模板**（直接用）：

```bash
copilot -p "
我正在开发 E-Kanban 项目，基于 Vol.Core 框架，SqlSugar ORM。

开发规范：
$(cat docs/kb-code-patterns.md)

现有代码资产：
$(cat docs/kb-code-assets.md)

需求：
【这里粘贴需求】

现有代码示例（参考风格）：
$(cat VOL.Entity/SystemModels/User.cs)

请帮我生成代码，放在正确的位置，遵循现有代码风格。
" --allow-all-tools
```

### 批次完成后做什么

一个批次的所有任务都完成后：

1. 整体编译一次项目
2. 确保没有编译错误
3. 确认所有任务状态都已标记为完成
4. 提交代码到 Git
5. 休息一下，然后开始下一个批次

**提交示例**：
```bash
git add .
git commit -m "feat: complete first batch - infrastructure layer"
git push
```

### 阶段三：需求完成后的更新

**这是非常重要的一步！** 需求开发完成后，必须更新相关文档。

#### 步骤一：更新 README.md

**为什么要做这一步？**
- 让其他团队成员了解新功能
- 保持项目文档的时效性
- 便于后续维护和交接

**怎么做？**
1. 打开项目根目录的 `README.md`
2. 在"核心功能"部分添加新功能说明
3. 在"开发进度"部分更新状态
4. 添加新功能的配置说明（如果有）

**示例**：
```markdown
## ✨ 核心功能

| 功能 | 说明 |
|------|------|
| **新功能名称** | 功能描述 |

## 🧩 开发进度

- [x] **v1.4 新增**: 新功能名称 - 功能描述
```

#### 步骤二：更新代码资产盘点报告

**为什么要做这一步？**
- 保持代码资产盘点报告的准确性
- 让团队成员了解最新的代码结构
- 便于后续需求开发时参考

**怎么做？**
1. 打开 `docs/kb-code-assets.md`
2. 在相应章节添加新增的模块信息：
   - 新增的实体类
   - 新增的服务接口和实现
   - 新增的仓储接口和实现
   - 新增的控制器
   - 新增的前端组件
3. 更新模块依赖关系图（如果有）

**示例**：
```markdown
### 新增模块

#### 实体类
- `VOL.Entity/DomainModels/NewEntity.cs` - 新实体描述

#### 服务层
- `EKanban/IServices/INewService.cs` - 新服务接口
- `EKanban/Services/NewService.cs` - 新服务实现
```

#### 步骤三：更新需求代码映射表

**为什么要做这一步？**
- 保持需求与代码的映射关系
- 便于后续查找和修改
- 形成完整的开发记录

**怎么做？**
1. 打开 `docs/kb-module-index.md`
2. 添加新需求与代码的映射关系
3. 记录新增的文件和修改的文件

**示例**：
```markdown
### 新需求名称

| 需求点 | 实现位置 | 文件路径 |
|--------|---------|----------|
| 功能点1 | 实体类 | `VOL.Entity/DomainModels/NewEntity.cs` |
| 功能点2 | 服务层 | `EKanban/Services/NewService.cs` |
| 功能点3 | 控制器 | `VOL.WebApi/Controllers/EKanban/NewController.cs` |
```

#### 步骤四：编写知识沉淀文档

**为什么要做这一步？**
- 记录开发过程中的经验和教训
- 为后续开发提供参考
- 形成团队知识库

**怎么做？**
1. 打开 `src/feature-records/需求名称/06-知识沉淀.md`
2. 记录以下内容：
   - 开发过程中遇到的问题
   - 解决方法和思路
   - 最佳实践和经验总结
   - 后续改进建议

**示例**：
```markdown
# 知识沉淀

## 开发经验

### 技术选型经验
- 选择 XXX 技术的原因
- 遇到的问题和解决方法

### 最佳实践
- 代码规范
- 性能优化技巧

## 问题与解决方案

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 问题描述 | 原因分析 | 解决方法 |
```

#### 步骤五：代码提交与版本管理

**怎么做？**
1. 提交所有代码和文档到 Git
2. 编写详细的提交信息，说明本次改动的内容
3. 使用语义化提交信息格式：
   ```
   feat: 添加新功能
   fix: 修复 bug
   docs: 更新文档
   refactor: 重构代码
   test: 添加测试
   chore: 构建/工具链相关
   ```
4. 为重要的里程碑创建 Git 标签
5. 将开发分支合并到主分支

**提交示例**：
```bash
git add .
git commit -m "feat: 添加新功能名称

- 添加新实体类 NewEntity
- 添加新服务 NewService
- 添加新控制器 NewController
- 更新 README.md
- 更新 kb-code-assets.md
- 更新 kb-module-index.md"
git push
git tag -a v1.4.0 -m "Release v1.4.0: 新功能名称"
git push origin v1.4.0
```

## ❓ 遇到问题怎么办

### 使用知识库解决问题

**步骤 1：查问题库**
```
打开 kb-problem-solutions.md → 搜索关键词 → 找到类似问题和解决方案
```

**步骤 2：查相似模块**
```
打开 kb-module-index.md → 找到相似功能 → 阅读代码实现
```

**步骤 3：查代码模式**
```
打开 kb-code-patterns.md → 找到对应的代码模板 → 参考实现
```

### 常见问题快速索引

| 问题类型 | 解决方案 | 参考文档 |
|---------|---------|---------|
| **需求不清晰** | 回看需求分析文档，或澄清需求 | `src/feature-records/需求名称/01-需求分析.md` |
| **不知道放哪里** | 查模块索引 | [kb-module-index.md](./kb-module-index.md) |
| **代码不编译** | 查错误信息，参考相似模块 | [kb-problem-solutions.md](./kb-problem-solutions.md) |
| **不知道怎么实现** | 查相似模块的代码 | [kb-module-index.md](./kb-module-index.md) |
| **代码风格不对** | 查代码规范 | [kb-code-patterns.md](./kb-code-patterns.md) |
| **遇到技术难题** | 查问题库 | [kb-problem-solutions.md](./kb-problem-solutions.md) |

### 使用 Copilot 解决问题

```bash
copilot -p "
我遇到了编译错误：
【粘贴错误信息】

这是相关代码：
【粘贴代码】

请帮我分析原因并修复。
" --allow-all-tools
```

### 记录新问题

解决问题后，记得记录到知识库：

1. 打开 `kb-problem-solutions.md`
2. 添加新的问题和解决方案
3. 或者打开 `src/feature-records/需求名称/06-知识沉淀.md` 记录

**记录模板**：
```markdown
## 问题：【问题描述】

### 原因分析
【为什么会发生】

### 解决方法
【怎么解决的】

### 代码示例
【关键代码片段】

### 预防措施
【以后怎么避免】
```
