# 结构化需求：E-Kanban 功能扩展 - 多项目支持和细粒度进度跟踪

---

## 1. 原始需求

用户需求原文：

> 1，2点可以重构，然后3，我想的是，不只是状态，每一个任务，我都要看到现在进度怎么样了，他在分析步骤还是需求步骤，现在添加了哪些文件之类的这些，或者说有没有任务在跑的日志跟踪，目前有吗？请你再研究代码回答我
>
> 请你把我的需求写一份，然后新开一个分支，把这三个需求，设置一下定时任务，让任务自动执行。请你不要自己执行，而是告诉任务如何做？

具体三个需求点：

1. **需求 1**：支持任务指派到不同项目（每个任务可以对应不同的外部项目）
2. **需求 2**：支持配置项目的本地文件位置和 Git 连接管理
3. **需求 3**：细粒度进度跟踪 - 不仅看到任务整体状态，还能看到：
   - 当前走到 S/E 六阶段的哪一步（需求分析 / 代码盘点 / 映射 / 开发 / 验证 / 沉淀）
   - 该任务当前已经添加/修改了哪些文件
   - 完整的执行日志跟踪，可追溯每一步操作

---

## 2. 核心目标

解决什么痛点：

| 痛点 | 解决方案 |
|------|----------|
| 目前 E-Kanban 只支持单个项目，无法管理多个独立项目的开发任务 | 新增项目仓库实体，支持每个任务关联不同项目 |
| 无法为每个项目配置独立的本地工作目录和 Git 远程地址 | 在项目实体中存储本地路径 + Git 地址 + 默认分支 |
| 只能看到任务整体状态（InProgress/Completed），看不到具体走到哪个开发阶段 | 新增阶段进度跟踪实体，记录每个阶段的状态和进度 |
| 无法追溯一个任务具体修改/新增了哪些文件 | 新增文件变更记录实体，跟踪每个文件的变更类型 |
| 缺少细粒度执行日志，出问题难以追溯 | 每个阶段都支持记录日志，所有执行结果都保留 |

**最终目标**：让 E-Kanban 支持多项目管理，并且提供从任务到阶段到文件的全链路可追溯能力。

## 3. 功能点拆解

### 3.1 功能点 1：新增 ProjectRepository 实体（项目仓库管理）

**需求描述**：
- 支持创建/编辑项目仓库
- 每个项目存储：项目名称、本地工作目录、Git 远程地址、默认分支、描述
- ExecutionCard 关联到 ProjectRepository（支持任务绑定到不同项目）

**数据表**：`ProjectRepositories`

### 3.2 功能点 2：新增 TaskPhaseProgress 实体（六阶段进度跟踪）

**需求描述**：
- 每个任务（ExecutionCard）拆分为六个阶段
- 跟踪每个阶段的状态：未开始 / 进行中 / 已完成
- 记录每个阶段的开始时间、完成时间
- 记录每个阶段的输出文档路径
- 记录每个阶段的执行日志

**阶段枚举**：
```
RequirementsAnalysis = 1  // 需求结构化分析
CodeInventory = 2        // 本地代码资产盘点
RequirementsCodeMapping = 3 // 需求-代码映射
IncrementalDevelopment = 4 // 增量开发实施
VerificationTesting = 5  // 验证测试
KnowledgeSettle = 6      // 知识沉淀
```

**数据表**：`TaskPhaseProgress`

### 3.3 功能点 3：新增 TaskFileChange 实体（文件变更跟踪）

**需求描述**：
- 记录每个任务变更的文件
- 支持标识变更类型：新增 / 修改 / 删除
- 记录文件路径
- 记录 Git commit hash（如果已提交）
- 记录变更时间

**数据表**：`TaskFileChanges`

### 3.4 功能点 4：更新现有服务层集成

**需求描述**：
- 更新 `ExecutionCard` 新增 `ProjectRepositoryId` 字段
- 在 AI 执行流程中，自动推进阶段状态
- 每次执行记录文件变更
- 提供查询接口：获取任务进度详情（当前阶段 + 文件变更 + 日志）

### 3.5 功能点 5：数据库脚本更新

**需求描述**：
- 生成三个新表的建表 SQL
- 添加外键关系
