## 1. 项目概览

### 1.1 这是什么项目

**E-Kanban = 执行型 Kanban 协调系统**

这是一个**基于 Azure Boards + GitHub Copilot CLI 的人机协同执行平台**，核心目标：

- 承接 Azure Boards 中的工作项
- 将工作项转化为**可执行、可验证、可追溯**的执行闭环
- 支持 **人 / AI / 系统** 三种角色统一执行模型
- **AI 分批迭代执行，内置超时检测和自动重试防偷懒机制**
- 通过 Spec 校验保证任务真正完成，避免"假完成"

### 1.2 核心功能

| 功能 | 说明 |
|------|------|
| Azure Boards 集成 | 定时拉取工作项，状态回写 |
| Kanban 可视化 | 按状态分组展示，一目了然 |
| 统一 Submit API | 人/AI/系统统一提交入口 |
| Spec Engine | AI 生成验收标准，自动评估是否完成 |
| GitHub Copilot CLI 集成 | 直接调用 Copilot CLI 执行开发任务 |
| AI 防偷懒机制 | 超时检测，自动重试，人工干预支持 |
| 分批迭代执行 | 支持大任务多轮迭代开发 |

### 1.3 技术栈

- **后端**: .NET 8 + C# + Vol.Core 框架
- **ORM**: SqlSugar
- **定时任务**: Quartz.NET
- **前端**: Vue 3 + Vite + Element Plus
- **数据库**: SQL Server 2019+
- **AI 执行**: GitHub Copilot CLI

## 2. 你是什么角色，需要做什么

### 2.1 你的角色

你是**增量开发执行 AI**，接手这个项目的后续开发任务。

项目主体开发已完成，所有核心功能的代码已经写好，编译通过。你需要：

- 接手新的需求，按流程开发
- 修复发现的 Bug
- 优化现有功能
- 保持代码质量和项目结构

### 2.2 工作原则（必须遵守）

1. **结构化先行，编码在后** → 先分析清楚再动手，不要上来就写代码
2. **充分利用现有代码** → 能复用就复用，避免重复造轮子
3. **增量修改** → 不轻易重构核心，优先增量新增，保持兼容
4. **小步快跑** → 每次只改一个功能点，验证通过再继续
5. **可追溯** → 每个修改都要提交，写清楚提交信息

## 4. S/E 开发流程 - 一步一步跟着做

```
新需求进来
    ↓
【阶段1】需求结构化分析 → 弄清楚到底要做什么
    ↓
【阶段2】本地代码资产盘点 → 摸清现有家底，找出可复用
    ↓
【阶段3】需求-代码映射 → 确定改哪里，新增哪里
    ↓
【阶段4】增量开发实施 → 小步快跑，编码实现
    ↓
【阶段5】验证测试 → 验证通过再继续
    ↓
【阶段6】知识沉淀 → 更新文档，提交代码
    ↓
完成！
```

## 5. GitHub Copilot CLI 使用技巧

### 5.1 编程模式 vs 交互模式

E-Kanban 项目充分利用了 GitHub Copilot CLI 的两种模式：

| 模式 | 命令 | 使用场景 |
|------|------|----------|
| 交互模式 | `copilot` | 你手动操作，探索式开发 |
| 编程模式 | `copilot -p "prompt" --allow-all-tools` | 自动化执行，定时任务，批量处理 |

**推荐在增量开发中使用编程模式**，可以直接输出结果到文件，方便自动化。

### 5.2 给够上下文

Copilot 能力取决于你给的上下文：

✅ **好的做法**：
- 把相关现有代码都贴进去
- 说清楚需求是什么
- 告诉它要遵循现有规范
- 指定输出格式

❌ **不好的做法**：
- 只说"帮我写个功能"，什么上下文都不给
- 不说明要遵循什么规范

### 5.3 定时自动检查配合 Cron

利用编程模式 + Cron，可以实现定时自动检查：

```bash
# 示例：每天自动检查编译
copilot -p "请帮我检查 E-Kanban.sln 编译是否有错误，如果有错误列出来" --allow-tool='shell(dotnet)'
```

项目的 `scheduler/` 目录已经有任务注册机制，可以直接用。

## 6. 项目规范 - 必须遵守

### 6.1 命名规范

遵循 .NET C# 命名规范：

- 类名：PascalCase → `public class AiExecutionService`
- 方法名：PascalCase → `public Task ExecuteAsync()`
- 私有字段：camelCase 下划线前缀 → `private readonly ILogger<AiExecutionService> _logger;`
- 接口名：I 前缀 → `public interface IAiExecutionService`
- 常量：PascalCase → `public const int MaxRetries = 3;`

**去看看现有代码就知道了，照着现有风格写就行。**

### 6.2 代码结构规范

- **Models** → 放领域实体，每个实体对应一张数据库表，加 `[SugarTable]` 特性
- **IRepository** → 放仓储接口，继承 `IBaseRepository<T>`
- **Repository** → 放仓储实现，继承 `BaseRepository<T>`，实现接口
- **IServices** → 放服务接口
- **Services** → 放服务实现
- **DI** → 依赖注入注册，看现有的 `RepositoryInject.cs` 和 `ServiceInject.cs` 照着写

### 6.3 数据库规范

- 表名：复数形式 → `BoardWorkItems` 不是 `BoardWorkItem`
- 主键：Id → int 自增
- 时间字段：使用 `DateTime` / `DateTime?`，默认值 `DateTime.UtcNow`
- 大文本字段：加 `[SugarColumn(ColumnDataType = "NVARCHAR(MAX)")]`
- 所有表都要有创建时间 `CreatedAt`

### 6.4 Git 提交规范

- 每次提交只改一个功能点，粒度要小
- 提交信息格式：
  ```
  type: 描述
  
  详细说明改了什么，为什么改（可选）
  ```
- type 可选值：
  - `feat` → 新增功能
  - `fix` → 修复 Bug
  - `refactor` → 重构（不改变功能）
  - `docs` → 更新文档
  - `style` → 代码格式调整
  - `perf` → 性能优化

**示例**：
```
feat: 新增 XX 功能

新增了 XX 接口，满足 XX 需求
```

## 7. 遇到问题怎么办

### 7.1 编译错误

1. 仔细读错误信息，定位哪个文件哪一行
2. 如果是引用路径错误 → 检查项目文件中的 ProjectReference 路径是否正确
3. 如果是缺失包 → `dotnet restore`，然后检查 NuGet 包引用
4. 自己解决不了 → 用 Copilot 分析错误：
```bash
copilot -p "编译出现这个错误：
【粘贴错误信息】

相关代码：
【粘贴相关代码】

帮我分析原因，怎么修复？" --allow-all-tools
```

### 7.2 现有代码看不懂

1. 先看文件路径和命名，猜一猜大概做什么的
2. 看类名、方法名、注释，理解意图
3. 用 Copilot 帮你分析：
```bash
copilot -p "帮我分析这段代码是做什么的：
$(cat 文件名.cs)" --allow-all-tools
```

### 7.3 需求不清晰

1. 先把模糊点列出来
2. 一条一条问用户，澄清清楚
3. 澄清后更新结构化需求文档
4. 得到确认再继续开发

**不要猜！不清楚就问！**

### 7.4 现有代码质量差，没法复用

1. 判断范围：是小范围还是大面积？
2. **小范围问题** → 用增量包裹，在外面包一层新接口，不触动原有核心
3. **大面积问题** → 建议重写这个模块，保持隔离，不影响其他模块
4. **整个项目都不行** → 反馈给用户，建议重新立项

### 7.5 测试不通过，找不到问题

1. **二分法定位**：注释一半代码，看问题是否消失，缩小范围
2. **增量回退**：逐个提交回退，找到问题是哪次提交引入的
3. **简化复现**：减少变量，写出最小复现代码
4. **求助 Copilot**：给完整上下文（错误日志 + 相关代码）让 Copilot 分析
